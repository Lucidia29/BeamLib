Rotation transform
S·λ·S
What Batch 3 adds. 3D Euler–Bernoulli beam with 6 DOFs/node [ux, uy, uz, θx, θy, θz]. Two uncoupled bending planes (xy uses EIz, xz uses EIy) + St. Venant torsion (GIx/L) + required torsional rotary inertia (ρIx). Local frame built via Gram–Schmidt projection of a per-element refVector, with a deterministic fallback ladder for parallel cases. 3D rotation transformation uses S·λ·S with S = diag(1, −1, 1) to correctly map BeamLib's structural-convention rotation DOFs across frames.
Codex Round 1 (Batch 3) blockers resolved. Codex's review of the initial Batch 3 commit flagged two issues: (1) the rotation block of the 12×12 transformation matrix was λ instead of S·λ·S, which silently failed any geometry where λ commutes with S (e.g. beams in the xz-plane); (2) the EB3D mass-matrix path lacked direct test coverage. Both were addressed: the rigid-body patch test test_eb3d_rigid_body first confirmed the bug (residual norm ≈ 43140 on the buggy code), then verified the fix (residual ≈ 1.7×10⁻¹⁰). The mass-matrix path is now covered by test_eb3d_mass. See REVIEW_NOTES.md "CC Response to Codex Review — Batch 3 EB3D".

Why λ alone is wrong

BeamLib stores rotation DOFs in mixed convention:

  θx = +Ωx (RH)

  θy = −Ωy (structural, opposite RH)

  θz = +Ωz (matches RH)

In any frame:

  θ = S · Ω,   S = diag(1, −1, 1)

For a physical rotation pseudovector,

  Ωlocal = λ · Ωglobal

So the stored rotation transforms as:

  θlocal = S · Ωlocal = S · λ · Ωglobal

  = S · λ · S · (S · Ωglobal)

  = (S · λ · S) · θglobal

Plain λ is correct only when λ commutes with S (e.g. xz-plane beams). For any other orientation, λS − Sλ ≠ 0 and the rigid-body strain has a spurious term.

Implementation. BeamMath3D::buildTransformation3D applies S·λ·S to the two rotation 3×3 blocks of the 12×12 element transformation, leaving the translation blocks as plain λ. The S adapter is a row-1 and column-1 sign flip on λ: lambda_rot.row(1) *= -1.0; lambda_rot.col(1) *= -1.0;. The fixed transformation preserves orthogonality (S² = I) and reproduces the correct rigid-body kinematics for every orientation tested.

Why ρIx is mandatory. Without the torsional rotary inertia block on the θx DOFs, the polar inertia is absent and torsional eigenmodes either vanish or appear at spurious zero frequency in modal analysis. PROJECT_SPEC §3.4 and Codex Round 3 issue 2 both require this for any EB3D model that goes into Batch 8/9. The bending rotary inertia (ρIy, ρIz) is intentionally absent for EB and will be added by Timoshenko in Batches 4/5.
